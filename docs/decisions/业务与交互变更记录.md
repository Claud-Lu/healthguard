# 业务与交互变更记录

本文件记录 HealthGuard 开发过程中的关键范围、交互、接口和架构决策。每次变更都要写清背景、决策、影响范围、验证方式和待确认事项。

## 2026-05-20: MVP 范围收窄

### 背景

原始技术方案覆盖 H5、小程序、Android、iOS、Kafka、ClickHouse、MySQL、Redis、MinIO、Grafana、符号化服务和 Session Replay，更接近 6 人团队 3 个月的完整企业级项目。

### 决策

MVP 阶段只做 H5 和微信小程序，先打通采集、入库、聚合、查询、dashboard 展示闭环。Android、iOS、ANR、符号化、Kafka、Redis、MinIO、Grafana、Session Replay 延后。

### 影响范围

- README 和技术方案改为“终局愿景 + MVP 路线”双层结构。
- 首批实现目录优先考虑 `packages/sdk-web`、`packages/sdk-miniprogram`、`apps/server`、`apps/dashboard`、`examples/vue3-demo`。
- 第一阶段验收以端到端可见错误为准。

### 验证方式

- H5 demo 抛错后，dashboard 能看到错误。
- 失败请求能记录 URL、method、status、duration。
- 相同错误能聚合为同一个 issue。

### 待确认事项

- MVP 后端最终采用 Node.js/Fastify 还是 Go 单体。
- 元数据存储采用 SQLite、PostgreSQL 还是 MySQL。

## 2026-05-20: 管理台技术栈建议

### 背景

原方案写的是 React 18 + Ant Design + Zustand。考虑长期维护成本和轻量迭代节奏，Vue 3 技术栈更贴合。

### 决策

MVP 管理台建议使用 Vue 3 + Vite + Element Plus + Pinia + Axios + ECharts。

### 影响范围

- 后续 `apps/dashboard` 初始化时按 Vue 技术栈创建。
- UI 开发必须遵守项目内设计规范，避免通用模板感。

### 验证方式

- Dashboard 页面能在本地启动。
- 核心页面在桌面和移动宽度下无明显布局重叠。

## 2026-05-20: Phase 1 技术骨架初始化

### 背景

项目从文档骨架进入开发阶段，需要先建立可测试的 monorepo、事件协议、H5 SDK 最小能力和 collector 聚合入口。

### 决策

- MVP 后端采用 Node.js + Fastify + TypeScript。
- 包管理使用 yarn workspace。
- 事件协议先沉到 `packages/core`，供 SDK、server 和后续 dashboard 复用。
- `apps/server` 第一阶段使用内存存储，先验证采集、校验、聚合和查询闭环；持久化存储留到 H5 端到端跑通后替换。
- `packages/sdk-web` 第一阶段提供手动 `captureException`、`captureHttp`、breadcrumb 队列和 batch flush，浏览器全局自动 hook 后续迭代补齐。

### 影响范围

- 新增 `packages/core`、`packages/sdk-web`、`apps/server`。
- 新增 root `package.json`、TypeScript、Vitest、ESLint 配置。
- README 更新当前代码结构和常用命令。

### 验证方式

- `yarn test` 覆盖事件 schema、敏感 query 脱敏、fingerprint 稳定性、SDK 队列 flush、collector issue 聚合。
- `yarn type-check` 验证 TypeScript 类型。
- `yarn lint` 验证基础代码规范。

### 待确认事项

- 持久化层优先 SQLite、PostgreSQL 还是 ClickHouse + 关系库组合。
- H5 SDK 下一步是否先做 `window.onerror` / `unhandledrejection` 自动 hook，还是先完成 Vue3 demo 和 dashboard 查询页。

## 2026-05-20: README 中英文拆分与 H5 自动采集起步

### 背景

项目面向开源场景，需要英文 README 作为默认入口，同时保留中文 README 方便后续中文协作和 AI 接力。Phase 1 下一步需要让 `sdk-web` 从手动调用走向真实浏览器自动采集。

### 决策

- 保留 `README.md` 作为英文版入口。
- 新增 `README.zh-CN.md` 作为中文版，两个 README 顶部互相链接。
- `packages/sdk-web` 新增 `autoCapture` 选项，支持自动监听 `error`、`unhandledrejection` 和 `fetch`。
- 自动 fetch 采集只记录 method、URL、status、duration、success 和错误信息，不采集 request body / response body。

### 影响范围

- README 维护变为中英文双文件，后续重要进度需要同步更新两个入口。
- H5 SDK 已具备下一步接入 `examples/vue3-demo` 的基础。

### 验证方式

- `packages/sdk-web/src/index.test.ts` 覆盖 window error、unhandled promise rejection 和 failed fetch 自动采集。
- 继续使用 `yarn test`、`yarn type-check`、`yarn lint`、`yarn build` 做阶段验收。

## 2026-05-20: MVP 本地闭环完成

### 背景

Phase 1 需要从 SDK 采集、collector 聚合、demo 触发到 dashboard 可视化形成可验证闭环，后续才能在真实业务项目中试接入。

### 决策

- `sdk-web` 补齐资源错误、fetch、XHR、失败重试和手动性能事件。
- SDK 自动 HTTP 采集必须排除 collector endpoint，避免 SDK 自己的上报请求被递归采集。
- `apps/server` 增加应用管理、概览指标、issue 详情和最近事件查询。
- `examples/vue3-demo` 用浏览器事件分发模拟 JS / Promise 错误入口，避免 Vite dev overlay 阻挡验证。
- `apps/dashboard` 用轻量 Vue + Vite 实现 MVP 管理台，不在首版引入复杂 UI 依赖。
- `packages/sdk-miniprogram` 先提供最小微信小程序 SDK 包，为后续真实业务小程序试接入做准备。

### 影响范围

- 新增 `apps/dashboard`、`examples/vue3-demo`、`packages/sdk-miniprogram`、`scripts/dev-local.sh`。
- README 中英文版同步更新当前 MVP 结构和本地启动命令。
- 真实业务接入测试应使用测试 collector 地址或本地地址，不直接接生产采集服务。

### 验证方式

- `yarn test` 覆盖 20 个单元测试。
- 本地启动 collector、demo、dashboard 后，浏览器点击 demo 的 JS 错误、Promise 异常、失败请求，再刷新 dashboard，能看到 3 个事件、2 个错误、1 个失败请求和 issue 详情。

## 2026-05-21: Dashboard 本地认证与中英文切换

### 背景

Dashboard 首版以单页监控为主，默认英文展示，且项目 app key 更像示例配置。私有部署场景需要先完成本地注册 / 登录，再让不同使用者维护自己的项目列表；开源项目也需要避免只维护单一语言界面。

### 决策

- Dashboard 增加本地注册、登录、当前用户识别和退出登录。
- `apps/server` 的 `/api/apps` 改为需要登录后访问，并按用户隔离项目列表。
- 创建项目时必须选择应用类型，支持 `web`、`wechat-miniprogram`、`alipay-miniprogram`、`flutter`、`other`，生成的 app key 使用类型前缀。
- Dashboard 增加英文 / 中文切换，首次进入时根据浏览器系统时区判断默认语言：中国相关时区默认中文，其他时区默认英文。
- 后续涉及 Dashboard 可见文案或 README 的改动，必须同步维护英文和中文版本。

### 影响范围

- `apps/server` 增加本地用户、会话和按用户隔离的项目管理。
- `apps/dashboard` 增加认证页、语言切换、项目列表和创建项目表单。
- `README.md` 与 `README.zh-CN.md` 同步补充认证、项目类型、app key 和国际化说明。
- 当前 MVP 仍使用内存存储；服务重启后用户、会话、项目和事件都会清空，持久化数据库是下一阶段部署增强事项。

### 验证方式

- 单元测试覆盖注册、登录、当前用户、项目鉴权、用户隔离、项目类型 app key 和语言默认值。
- 阶段验收继续执行 `yarn test`、`yarn type-check`、`yarn lint`、`yarn build`。
- 本地或测试环境冒烟验证注册 / 登录后可以创建项目，并在项目列表中看到对应类型和 app key。

### 待确认事项

- 私有部署的元数据持久化优先使用 SQLite、PostgreSQL 还是 MySQL。
- 是否增加管理员初始化流程、密码重置和会话过期策略。

## 2026-05-21: 操作文档双轨维护

### 背景

HealthGuard 后续会频繁由人工和 AI CLI 接力操作。只给人看的说明不足以支撑 AI agent 稳定执行；只给 AI 的命令清单又不利于人工理解产品使用流程。

### 决策

- 新增 `docs/operator-guide.md` 作为人工操作文档。
- 新增 `docs/ai-cli-operations.md` 作为 AI CLI / agent 操作手册。
- 登录、注册、项目创建、app key、SDK 接入、部署、持久化或验证流程变化时，必须同步修改这两份文档。
- README 中英文版和 handoff prompt 都加入这条同步规则。

### 影响范围

- 后续操作流程不再只写在 README 或对话里。
- AI agent 执行项优先从 `docs/ai-cli-operations.md` 读取命令、边界和验证步骤。
- 开源仓库文档继续只保留通用操作，不记录业务项目名、公司域名、IP、个人路径、密钥或业务专用 app key。

### 验证方式

- 文档变更后检查 README 中英文、人工操作文档、AI CLI 操作文档和决策记录是否同步。
- 涉及代码行为变化时继续执行 `yarn test`、`yarn type-check`、`yarn lint`、`yarn build`。

## 2026-05-21: Dashboard 首页改为项目列表

### 背景

Dashboard 原本使用 `demo-web` 作为默认 app key，登录后刷新会直接展示 `demo-web` 的单项目数据。这会让用户误以为首页绑定了某个真实项目，也不符合“先看项目列表，再进入项目详情”的使用模型。

### 决策

- 登录后的 Dashboard 首页只展示当前用户的项目列表。
- 刷新时只拉取项目列表，不再默认请求 `demo-web` 或其他单个项目数据。
- 点击项目卡片或侧边栏项目后，才进入该项目详情并请求 overview、issues 和 SDK snippet。
- 创建项目后可以直接进入新项目详情，方便复制 app key 和接入 SDK。

### 影响范围

- `apps/dashboard` 的默认选中项目从 `demo-web` 改为空态。
- `docs/operator-guide.md` 与 `docs/ai-cli-operations.md` 同步补充项目列表首页和点击进入详情的操作方式。

### 验证方式

- 登录后刷新页面应停留在项目列表页。
- 项目列表页不展示 `demo-web` 指标。
- 点击项目后才展示该项目的指标、Issue 和 SDK 接入片段。
- 继续执行 `yarn test`、`yarn type-check`、`yarn lint`、`yarn build`。
